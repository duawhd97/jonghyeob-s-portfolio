리테일 IoT 플랫폼 SSP

솔루엠 2025.4 - 2026.2
서비스 기획 프로젝트 매니징 데이터 구조 설계 플랫폼 설계
기여도 100%

Overview

SSP는 글로벌 리테일 매장에서 사용하는 다양한 IoT 디바이스를 통합 관리하기 위한 솔루엠의 차세대 플랫폼입니다. ESL, Gateway, Vision Camera, Sensor 등 매장 내 디바이스 데이터를 기반으로 매장 운영 상황을 모니터링하고 문제 발생 시 빠르게 대응할 수 있는 운영 환경을 구축하는 것을 목표로 했습니다.

또한 레거시 시스템인 AIMS 운영을 유지하면서도 플랫폼 확장이 가능하도록 구조를 설계해야 하는 미션이 있었습니다. 기존 AIMS의 구조를 분석하고, 매장 운영 관점에서 Store Dashboard와 Digital Twin(Map View) 기능을 기획했습니다.

Problem

기존 AIMS 플랫폼의 구조적 한계:

  • 디바이스 관리 중심의 플랫폼 구조 — ESL, Gateway, Vision, Sensor 등 디바이스별 관리 기능에 초점이 맞춰져 매장 운영 관점에서 전체 상황을 파악하기 어려웠습니다. 고객사 납품 시마다 별도 대시보드를 추가 개발해야 하는 상황이 반복되었습니다.
  • 디바이스 위치 정보 관리 구조 부재 — 매장 내 장애 발생 시 문제 위치를 직관적으로 파악하기 어려운 구조. 글로벌 리테일 매장의 경우 위치 기반 모니터링이 필수적이었습니다.
  • Vision 기반 재고 감지 정확도 문제 — 약 30% 수준의 오판 발생. 배터리 수명 문제로 4시간 단위 스케줄 기반 촬영 방식을 사용해 실시간 감지가 어려웠습니다.

Analysis

기존 플랫폼의 API 구조와 데이터 모델을 분석하고 매장 운영 관점에서 필요한 정보 구조를 재정의했습니다.

기존 데이터 구조:

Customer → Store → Shelf → Device

이 구조는 디바이스 관리에는 문제가 없었지만, 글로벌 리테일 매장의 층(Floor), 구역(Zone) 등 매장 공간 정보를 체계적으로 표현할 수 없어 공간 기반 모니터링 기능 구현이 어려운 구조였습니다.

Solution

1. Store Dashboard 설계

매장 운영자가 디바이스 상태와 매장 이벤트를 한눈에 파악할 수 있는 운영 중심 대시보드 구조를 설계했습니다.

  • ESL 상태 요약
  • Gateway 연결 상태
  • 디바이스 오류 발생 현황
  • 매장 운영 이벤트 알림
  • 우선 조치 필요 항목

2. Digital Twin 기반 Map View

기존 데이터 모델을 확장하여 위치 기반 데이터 구조를 설계했습니다.

Customer → Store → Floor → Zone → Shelf → Device

이 구조를 기반으로 매장 공간 위에서 디바이스 상태와 이벤트를 확인할 수 있는 Digital Twin 기반 Map View 기능을 기획했습니다.

3. Vision + Load Cell 이중 검증 구조

기존 Vision 카메라의 오판 문제를 해결하기 위해 Load Cell(무게 센서) 기반 이중 검증 구조를 제안했습니다.

  • Load Cell 센서가 상품 무게 변화를 감지
  • 특정 임계값 변화 발생 시 Vision 카메라 촬영 트리거
  • 이미지 분석 결과와 무게 데이터를 교차 검증

Key Contribution

1. 레거시 시스템 구조 분석

기존 AIMS 플랫폼의 API와 데이터 구조를 분석하여 디바이스 관리 중심 구조의 한계를 파악하고 매장 운영 관점에서 필요한 기능을 도출했습니다.

2. 위치 기반 데이터 모델 설계

Customer → Store → Floor → Zone → Shelf → Device 구조로 확장하여 매장 공간 기반 디바이스 관리가 가능한 데이터 모델을 설계했습니다.

3. 매장 운영 중심 플랫폼 UX 구조 설계

Store Dashboard와 Digital Twin 기반 Map View 기능을 기획하여 디바이스 관리 중심 UI를 매장 운영 중심 모니터링 기능을 재설계했습니다.

4. IoT 센서 데이터 기반 이벤트 구조 설계

Vision Camera 재고 감지 오판 문제를 분석하고 Load Cell 센서를 활용한 이벤트 기반 감지 구조를 제안했습니다.

5. 글로벌 협업 환경에서의 요구사항 조율

한국 유지보수 조직, 인도 개발팀, 싱가포르 마케팅 조직과 협업하며 기술적 제약과 비즈니스 요구사항을 조율했습니다.

AI 기반 프로토타입 검증 프로세스

솔루엠 2026.2
프로세스 개선 서비스 기획
기여도 100%

Overview

생성형 AI 설계 도구를 활용하여 데모 애플리케이션을 빠르게 제작하는 과제에서 시작된 프로젝트입니다. 진행 과정에서 기존 서비스 개발 프로세스의 의사결정 리드타임이 길어지는 문제를 발견하고, PRD 기반 워킹 프로토타입을 생성하여 기능을 검증하는 새로운 개발 프로세스를 실험했습니다.

Problem

기존 프로세스:

요구사항 정의 → 와이어프레임 설계 → 디자인 제작 → 개발 전달
  • 의사결정 리드타임 증가 — 이해관계자 리뷰가 여러 차례 반복되며 기획 검증 단계에만 평균 4주 이상 소요
  • 실제 동작 검증의 어려움 — 와이어프레임/디자인 시안으로는 인터랙션 오류, 사용자 흐름 문제, 기능 누락 등 검증이 어려움
  • 반복적인 수정 비용 발생 — 기획 오류가 개발 단계 이후 발견되어 기획-디자인-개발 수정이 반복

Experiment

Test A — 요구사항 직접 입력

상위기획 부서의 요구사항을 그대로 AI 설계 도구에 입력. 요구사항 해석 오류, UI 구조 일관성 부족, 시스템 컨텍스트 반영 한계가 발생했습니다.

Test B — PRD 기반 입력

요구사항을 PRD 형태로 재구조화하여 입력. 서비스 목적, 사용자 시나리오, 기능 정의, 화면 구조, 인터랙션 흐름, 기술적 제약을 포함시켰습니다.

Solution

PRD 기반 입력 방식이 더 안정적이라는 결과를 기반으로 프로세스를 재정의했습니다.

기존:

요구사항 정리 → 와이어프레임 제작 → 디자인 제작 → 개발 전달

개선:

PRD 작성 → AI 기반 워킹 프로토타입 생성 → 동작 검증 → 개발 전달

Result

최종 합의 도달
4주 → 1~2주
의사결정 리드타임 대폭 단축
수정 반영 속도
5일 → 1~2일
프로토타입 기반 빠른 수정

Key Contribution

1. AI 설계 도구 활용 방식 실험 설계

Load Cell 기반 데모 애플리케이션 제작 과정에서 생성형 AI 활용 방식을 비교하기 위해 Test A(요건 문서 그대로 입력)와 Test B(PRD 재구조화 후 입력) 방식의 실험을 설계하고 결과 품질 차이를 분석했습니다.

2. PRD 기반 입력 구조 정의

AI 설계 도구의 결과 품질을 개선하기 위해 요구사항을 PRD 형태로 재구조화하여 입력하는 방식을 정의하고, 기능 요구사항·사용자 시나리오·인터랙션 흐름을 포함한 입력 구조를 설계했습니다.

3. 워킹 프로토타입 제작 및 결과 분석

Figma Make를 활용해 PRD 기반 워킹 프로토타입을 생성하고 생성 시간과 프롬프트 입력 회수를 비교 분석하여 AI 설계 도구 활용 시 생산성 향상 효과를 검증했습니다.

4. 개발 프로세스 개선안 설계

기존 '요구사항 정리 → 와이어프레임 → 디자인 → 개발 공유' 구조를 'PRD 입력 → 워킹 프로토타입 생성 → 동작 검증 → 개발 공유' 방식으로 재정의하여 기획 검증 리드타임을 단축할 수 있는 개발 프로세스를 설계했습니다.

5. 사업부 리더십 및 대표이사 대상 보고를 통해 프로세스 변경 공식화

AI 설계 도구 활용 실험 결과와 개발 프로세스 개선안을 사업부 리더십 및 대표이사 대상 보고로 공유하여 조직 내 AI 기반 설계 도구 활용 방향성과 실무 적용 기준 수립에 기여했습니다.

What I Learned

생성형 AI는 단순 작업 자동화 도구가 아닌, 기획과 개발 사이의 검증 단계를 앞당겨 의사결정 구조를 개선하는 도구로 활용될 수 있다는 점을 확인했습니다. 새로운 기술을 단순히 도입하는 것이 아니라 기존 업무 프로세스를 분석하고 조직에 맞는 방식으로 적용하는 것이 중요하다는 점을 배웠습니다.

현대엔지니어링 품질관리 포탈

서원정보 2023.8 - 2024.2
서비스 기획 프로젝트 매니징
기여도 70%

Overview

현대엔지니어링 현장의 품질 관리 업무를 지원하기 위해 통합 품질관리 포탈(Q-Portal)을 구축하는 SI 프로젝트입니다. 기존에는 현대건설과 현대엔지니어링이 서로 다른 품질 관리 시스템을 분리 운영하고 있어 데이터 연동 문제와 업무 비효율이 발생하고 있었습니다.

품질 관리 업무 프로세스 단순화, 품질 지식 데이터 활용 구조 설계, 법규 및 품질 기준 변화에 대응 가능한 시스템 구조 설계를 목표로 했습니다.

Problem

  • 품질 관리 시스템 분리 운영 — 시스템 간 데이터 연동 오류, 통계 분석 기능 오류, 동일 데이터 중복 관리
  • 비효율적인 보고서 작성 프로세스 — 자재 1개 → 공급업체 1개 → 보고서 1개 형태로, 공급업체 증가 시 보고서 작성 횟수 급증
  • 품질 관리 지식 데이터 활용 어려움 — 체크리스트가 프로젝트 단위로 분산 관리되어 재사용 불가
  • 법규 및 품질 기준 변화 대응 어려움 — 품질 시험 설정이 고정 구조로 설계되어 새로운 기준 대응 시 시스템 수정 필요

Solution

1. 자재 공급 승인 프로세스 개선

자재 1개 → 다수 공급업체 등록 가능 → 하나의 보고서에서 관리

2. Knowledge Data 공유 기능 설계

본사 품질 관리 담당자가 체크리스트를 설정하고, 현장 프로젝트에서 해당 데이터를 불러와 활용할 수 있는 구조를 설계. 품질 관리 지식이 조직 단위로 축적되는 구조를 구현했습니다.

3. 품질 시험 설정 확장

압축 강도 시험 최대 3회 → 6회로 확장하여 고층 건물 품질 시험 대응 및 향후 법규 변경 대응이 가능한 유연한 구조를 설계했습니다.

Result

운영 서버 반영 전후 2개월 데이터 비교 결과:

보고서 작성 건수
53% 감소
1,096건 → 515건
결재 반려 건수
90% 감소
63건 → 6건

Key Contribution

1. 품질 관리 업무 프로세스 분석

현장 업무 흐름을 분석하여 반복 작업과 데이터 오류의 구조적 원인을 정의했습니다.

2. 자재 공급 승인 프로세스 개선 기획

공급업체별 반복 보고서 구조를 단일 통합 보고서 구조로 전환하여 보고서 작성 건수 53% 감소에 기여했습니다.

3. Knowledge Data 공유 구조 설계

프로젝트 단위로 분산되던 체크리스트를 조직 단위로 재사용할 수 있는 구조를 기획하여 결재 반려 건수 90% 감소에 기여했습니다.

4. 품질 시험 설정 확장 기획

고정 3회 구조를 최대 6회로 확장하여 법규 변화 대응이 가능한 유연한 시스템 구조를 설계했습니다.

5. SI 프로젝트 산출물 작성 및 QA 참여

기능 정의서 작성과 QA 과정에 참여하여 기획 의도가 개발 결과물에 반영될 수 있도록 지원했습니다.

Skills

요구사항 분석 QA 관리 테스트 진행 관리 고객사 커뮤니케이션 문서화

CubeX PaaS 플랫폼

서원정보 2024.4 - 2024.11
서비스 기획 플랫폼 설계
기여도 60%

Overview

CubeX는 Kubernetes 기반의 클라우드 네이티브 PaaS 솔루션으로, 애플리케이션 배포와 운영을 효율적으로 관리할 수 있도록 지원하는 플랫폼입니다. 경북대학교 납품 계약을 기반으로 MVP 검증을 빠르게 진행해야 하는 상황에서 제품 초기 버전을 설계했습니다.

Problem

  • 복잡한 애플리케이션 배포 과정 — 컨테이너 이미지 관리, Deployment 설정, Pod 관리, 서비스 네트워크 설정 등 높은 Kubernetes 이해도가 필요
  • 운영 상태 모니터링의 어려움 — CLI 중심 환경에서만 애플리케이션 상태, Pod, 리소스 사용량 확인 가능
  • 애플리케이션 관리 기능의 분산 — 생성, 배포, 스케일링, 삭제 등 기능이 분산되어 사용성 저하

Solution

1. One-Step Application 생성 기능

UI 기반으로 애플리케이션 이름, 컨테이너 이미지, 리소스 설정, 네트워크 설정을 입력하여 바로 생성할 수 있도록 CLI 기반 설정 과정을 단순화했습니다.

2. Application Management 기능

애플리케이션 상태 확인, 실행 중인 Pod 확인, 수정, 삭제 등을 통합 관리할 수 있는 기능을 설계했습니다.

3. Auto Scaling 관리 기능

최소/최대 Pod 수, CPU 사용률 기준 등의 조건을 설정하여 애플리케이션 스케일링 정책을 관리할 수 있도록 설계했습니다.

Result

  • Kubernetes 기반 애플리케이션 운영 환경을 UI 기반 플랫폼 구조로 설계하여 CLI 중심 환경의 접근성 문제 해결
  • 기술 지식 없이도 플랫폼 운영이 가능한 사용자 중심 UX 흐름 구현
  • MVP 기반 빠른 검증을 통해 경북대학교 납품건으로 4억원 규모 계약 성사에 기여

Key Contribution

1. 클라우드 네이티브 플랫폼 UX 구조 설계

Kubernetes 기반 애플리케이션 운영 환경을 분석하여 플랫폼 UI 기반 관리 구조를 설계했습니다.

2. Application Lifecycle 기능 기획

애플리케이션 생성, 관리, 스케일링 등 Application Lifecycle 관리 기능을 정의했습니다.

3. MVP 기반 제품 기획 참여

경북대학교 납품 프로젝트 일정에 맞춰 MVP 중심의 제품 기능을 기획하고 플랫폼 초기 구조 설계에 참여했습니다.

Skills

MVP 기획 UX 설계 디자인 시스템 기획 기술 문서 작성 이해관계자 협업
